## 1. Таксономия: 4 типа клонов

Каноническая классификация (Bellon, Koschke; закреплена в обзоре Roy & Cordy
2009). На неё ссылаются все:

| Тип | Что это | Ось расхождения |
|-----|---------|-----------------|
| **Type-1** | идентичный код | только пробелы/комментарии |
| **Type-2** | структурно идентичный | переименованы идентификаторы, литералы, типы |
| **Type-3** | правленая копия («near-miss») | поверх Type-2 — вставлены/удалены/изменены строки |
| **Type-4** | семантический клон | функция та же, текст другой (другой алгоритм, `for`↔`while`) |

Тонкость из BigCloneBench: зона между Type-3 и Type-4 не бинарна — её дробят на
градации (Very Strongly / Strongly / Moderately / Weakly Type-3). Полезно, если
захотим шкалу уверенности вместо «клон / не клон».

## 2. Чем копипаст опасен — эмпирика, не мнение

Главный механизм вреда — **не дублирование само по себе, а рассинхрон при
правке**. Опорные результаты:

- **Inconsistent change → дефект.** Bettenburg et al. (WCRE'09): «почти каждое
  второе *непреднамеренное* несогласованное изменение клона приводит к багу».
  Это и есть «clone consistency-defect»: правишь одну копию, забываешь про
  сёстры. **Сильнейший сигнал вреда — это коэволюция копий по истории, а не их
  текстовое сходство в снапшоте.**
- Клонированный код в среднем bug-prone; правка метода с клонами дороже —
  стоимость растёт с долей системы, затронутой клон-группой.
- **Контр-данные есть.** Krinke; Harder & Göde («Cloned code: stable code»)
  показывают: клоны часто *стабильнее* обычного кода и сопровождаются
  согласованно. ⇒ вред **не универсален**, зависит от типа и паттерна клона.

## 3. Когда копипаст полезен — каталог Kapser & Godfrey

**«"Cloning considered harmful" considered harmful»** (Empirical Software
Engineering, 2008, 13(6):645–692). Доказывают: копипаст часто **осознанный
инженерный приём**. Их паттерны клонирования с оценкой авторов:

| Паттерн | Что это | Оценка |
|---------|---------|--------|
| **Verbatim snippets** | дословная копия куска | чаще **вредно** |
| **Parameterized code** | копия, параметризуемая в одну функцию | чаще **вредно** (это и есть наш TP) |
| **Boiler-plating** | повторяющийся структурный каркас (идиомы) | чаще **ок** |
| **Replicate & specialize** | копируют и специализируют под контекст | чаще **полезно** |
| **Cross-cutting concerns** | дублирование системной функциональности | чаще **полезно** |
| **Platform variations** | копии под ОС/платформу | приемлемо |
| **Language idioms** | идиомы языка | приемлемо |

Их тезис: решение о клоне требует **контекстного суждения**; самые проблемные
паттерны (verbatim, parameterized) надо устранять, остальные — чаще на пользу.

Это **внешняя опора нашему extractability-тесту**: вред определяется не текстовым
сходством, а тем, эволюционируют ли копии вместе и можно ли их осмысленно вынести.
Наши «worst-kind TP» (скопировал блок, поменял один токен) = их *parameterized
code*. Наши path-гарды (platform / perf / generated) = их *platform variations*,
*replicate & specialize*, плюс генерат (не клон вовсе — артефакт компилятора).

## 4. Формальный детект — по слоям представления

Конвейер у всех один: **препроцессинг → промежуточное представление → сравнение**.
Различаются представлением (Roy & Cordy 2009 — фреймворк сравнения):

| Подход | Представление | Берёт | Граница |
|--------|---------------|-------|---------|
| **Text / line** | сырые строки | Type-1 | слеп к переименованиям |
| **Token** (SourcererCC, CCStokener) | поток токенов | Type-1/2, быстро | слаб на near-miss без семантики |
| **AST** (Deckard) | дерево разбора | Type-1/2/3, устойчив к ins/del | кодирует синтаксис, не смысл |
| **PDG** | data/control-flow граф | Type-4, non-contiguous | дорого |
| **Metric / hybrid / ML** | признаки / эмбеддинги | Type-3/4 | recall ↑, цена ↑ |

Наш выбор — **token-based** (#056): дёшево, без libclang, ловит Type-1/2/3.
Type-4 сознательно вне scope fast-backend (нужен PDG/AST). Это ровно компромисс
«дешёвый широкий → дорогой точный» из литературы.

## 5. Gap-анализ: что из этого умеет наш код

Состояние кода на 2026-06-03 (`src/scan/duplication/`).

### 5.1. Определение **вида** копипаста — способность есть, но спит

`clone_classifier.{h,cpp}` → функция **`cloneType(a, b)`** уже считает:

| `cloneType()` | → канонический тип |
|---|---|
| `EXACT` | **Type-1** ✅ |
| `RENAMED` / `LITERAL` / `MIXED` | **Type-2** ✅ (даже дробнее канона: что именно правили) |
| `STRUCTURAL` | **Type-3** ✅ |
| — | **Type-4** ❌ — вне scope by design (нужен AST/PDG) |

**Главный разрыв: `cloneType()` нигде не вызывается в `scanForDuplication`.**
В `duplication_scanner.cpp` он только `#include`-ится; `Pair` поля типа не имеет,
отчёт тип не несёт. То есть классификатор реализован и покрыт тестами
(`tests/duplication_clone_classifier_test.cpp`), но **изолирован от пайплайна**.
Чтобы «определять виды» на выходе — нужно лишь добавить `type` в `Pair` и вызвать
классификатор в `phase3ScoreCandidates`. **Дёшево (~30 мин), чистая победа.**

Частично не хватает и дробного классификатора **характера правки** внутри Type-3
(EDITED-LOGIC = флип оператора vs EXPANDED/SHRUNK = доминируют `+`/`-`), описанного
в [../duplication_architecture.md](../duplication_architecture.md) §3.6 (#070 P3b).
`cloneType()` различает что правили (id vs lit), но не как менялась структура.

### 5.2. Определение **важности** копипаста — почти нет

Grep по `priority|severity|importance|rank` в коде дубликатов — **ноль совпадений**.
Важность как вычисляемый выход **не реализована**. Это согласуется с
mem:`fp_classification_rules`: «similarity-score ≠ refactor-priority, never conflate».

Сырьё для оценки важности в коде уже лежит, но в severity не сворачивается:

| Сигнал важности (из §2–3) | Есть в коде? | Чем |
|---|---|---|
| размер общего скелета | частично | `Fragment.tokenCount`, `lcs`-длина — не агрегируются в вес |
| логика vs boilerplate | частично | `Fragment.diversity` (trigram) — но используется как **FP-down-weight** (P1.1/P1.2), не как severity |
| число копий (3+) | **нет** | детектятся только **пары**; нет кластеризации в клон-классы (`wholeFileClones` — про другое) |
| extractability | **нет** | это пока человеческое суждение из правил, не код |
| benign vs harmful | бинарно | path-гарды P0.7-0.9 = suppress/keep, не градация важности |
| **inconsistent co-change** (сильнейший!) | **нет** | требует истории; #054 (diff-mode) ходит по git, но меряет дрейф, не co-evolution клон-группы |

**Вывод по важности:** код меряет **структурное сходство в снапшоте**, а
литература говорит, что главный предиктор вреда — **согласованность правок копий
во времени** (Bettenburg). Этого измерения нет. При этом инфраструктура близка:
diff-mode #054 уже умеет ходить по git-истории без checkout — это естественное
место, где co-change клон-группы можно посчитать. Самый ценный, но и самый
дорогой шаг.

### 5.3. Итог одной строкой

- **Виды** — код умеет (Type-1/2/3 через `cloneType`), но способность не выведена
  наружу; Type-4 — вне scope осознанно. → дешёвый wire-in.
- **Важность** — код **не умеет**: есть ингредиенты (`tokenCount`, `lcs`,
  `diversity`, индекс), нет модели severity и нет co-change сигнала. → требует
  дизайна; самый сильный сигнал (inconsistent co-change) живёт в плоскости #054.
