# Гайд: задачи для Haiku-исполнителя

Документ для двух читателей:

- **постановщик** (человек или старшая модель), который готовит задачу в `backlog/` под исполнение Haiku;
- **сам Haiku**, который задачу исполняет.

Происхождение: аудит задач #101/#102 (2026-06-11) перед передачей на исполнение.
Каждое правило здесь — не теория, а реальная найденная проблема.

---

## Часть 1 — постановщику: чек-лист Haiku-ready задачи

Haiku исполняет буквально и не достраивает контекст. Задача готова, только когда
все пункты ниже выполнены.

### 1.1. Ни одной неразрешённой развилки

Формулировки вида «если #X успел сделать Y — реюзай, иначе создай Z» — запрещены.
Постановщик обязан сам проверить состояние репо и записать **разрешённый факт с датой**:

> Плохо: «если #093 завёл `function_signature_scan`, расширить его; если нет — создать свой».
> Хорошо: «`function_signature_scan` не существует (проверено 2026-06-11) — создаём `function_body_scan` здесь».

### 1.2. Одна формула — один источник истины

Если семантика/формула менялась по ходу жизни задачи, пройтись по **всем** секциям
(Цель, Detection contract, План, Детальная инструкция, Следующие шаги, таблицы) и
выровнять. Haiku не выберет «более свежую» секцию — он возьмёт ту, которую читает.

Реальный случай: в #102 секция «Сама метрика» предписывала дефектную v1-формулу,
хотя «Следующие шаги» требовали v2. Устаревшую версию либо удалить, либо явно
пометить: «реализована, признана дефектной, не использовать».

### 1.3. Каждая ссылка на код — проверена живой

Перед передачей задачи прогнать каждый упомянутый файл/функцию через `ls`/`grep`.
Указывать точные имена API (`runGit(args, cwd)`, `collectAddedLines()`), а не
«helpers из git-слоя». Мёртвая ссылка для Haiku — повод написать дубликат.

### 1.4. У каждого контрольного кейса — ожидаемое число

Не «switch не должен взрываться», а таблица: «плоский 8-case switch → score **1**,
finding **НЕТ** (v1 давал 0→19)». Без чисел исполнитель не может self-check-нуться.

### 1.5. Меняется семантика → явно перечислить, какие ожидания тестов меняются

Самая опасная ловушка. Если старый тест кодирует **дефектное** поведение
(в #102: `switch_case_explosion` был `must_trigger`, что под правильной семантикой
неверно), а задача говорит «suite должен проходить», Haiku «починит» код обратно
под дефект. Обязательно писать: «ожидание кейса X сменить с A на B, потому что
старое ожидание было ошибкой; НЕ подгонять код под старое ожидание».

### 1.6. Артефакты задачи — только в репо

Никаких ссылок на `/tmp/...`, чужие машины, незакоммиченные локальные файлы.
Репро-кейсы, манифесты, ожидания — копировать в `experiments/` или `fixtures/`
до передачи задачи.

### 1.7. Известные ловушки реюзаемых инструментов — прямо в задаче

Если задача говорит «реюзай функцию X», постановщик обязан проверить её граничное
поведение и вписать ловушки. Реальный случай: `lex()` из token_normalizer вырезает
только `#if 0`-блоки, а `#if defined(X)` токенизирует — даёт фальшивый keyword `if`.
Без этой строчки в задаче Haiku получил бы тихо завышенные score.

### 1.8. Definition of done — проверяемый без суждений

«Все кейсы таблицы зелёные», «13/13 synthetic», «расхождение с clang-tidy ≤ 2 на
20 функциях», «0 нарушений собственных правил». Не «работает корректно».

### 1.9. Обязательное — императивом

«Можно», «желательно», «допустимо» Haiku читает как «необязательно». Для
обязательного — «обязан», «НЕ делать», «только так». Жирный шрифт и КАПС для
запретов работают.

## Часть 2 — Haiku: правила исполнения

### 2.1. Перед стартом — прочитать целиком

1. Задачу из `backlog/` — **всю**, включая «Не делать» и «Ключевые решения».
2. Все дизайн-доки, на которые задача ссылается.
3. [docs/code_style.md](../code_style.md) и [docs/code_quality.md](../code_quality.md).
4. [docs/decisions/](../decisions/) — прежде чем объявить отсутствующую фичу гэпом:
   возможно, это принятое решение, а не долг.

### 2.2. На что обращать внимание

- **Детальная инструкция в задаче авторитетна.** Нашёл противоречие между секциями
  задачи или между задачей и кодом — **остановись и спроси**, не выбирай молча.
- **Ожидаемые числа в таблицах — это контракт.** Каждый контрольный кейс прогоняй
  и сверяй с таблицей до объявления готовности.
- **Реюз перед написанием.** Перед созданием функции/файла — grep на похожее
  (задача обычно прямо называет, что реюзать). Нашёл похожее, но задача молчит —
  спроси, не дублируй.
- **Лимиты code_quality:** функции ≤ 30 строк, классы ≤ 300, ≤ 50 новых строк на
  коммит, ≤ 2 новых файла, 0 абстракций, которых не просили. Задача, требующая
  больше, явно об этом скажет.
- **Верификация:** сборка Debug (`cmake --build build/debug`), тесты
  (`ctest --output-on-failure`), lizard
  (`lizard --CCN 15 --length 30 --arguments 5 --warnings_only src/ include/ tests/`),
  dogfood (`./build/debug/src/archcheck` из корня → 0 нарушений на `src/`,
  `include/`, `tests/`). Пути в командах — только абсолютные.
- **Статусы честно:** упёрся во внешнее ограничение → статус `blocked` в
  `backlog/wip/`, НЕ `completed`. Completed = цель задачи достигнута и проверена.

### 2.3. Что НЕ делать никогда

- **Не коммитить без явной команды пользователя.** Закончил — собери, проверь, доложи и жди.
- **Не менять ожидание теста, чтобы он прошёл**, если задача явно не велела
  поменять именно это ожидание. Тест падает «неправильно» → остановись и доложи.
- **Не трогать контракты:** exit codes (0/1/2/3), CLI-флаги, `rules::Violation`,
  config schema — без прямого указания в задаче.
- **Не добавлять зависимости.** Стек заморожен: C++20, CMake, ryml, Catch2.
- **Не рефакторить соседний код** «заодно». Правки только в файлах из секции
  «Планируемые файлы» задачи; нужен файл сверх списка — спроси.
- **Не изобретать абстракции** (интерфейс «на вырост», generic engine, лишний слой) —
  это прямой forbidden pattern из code_quality.
- **Не трогать `backlog/pending/`** — ни читать в скоуп, ни упоминать в отчётах.
- **Не помечать задачу выполненной без прогона** build + tests + контрольных кейсов.
- **Не оставлять «outdated»-пометки** на документах: устаревший док либо
  переписывается, либо удаляется (по команде).
- **Не выходить за «Не делать»-секцию задачи.** Она такой же контракт, как и цель.

### 2.4. Формат доклада по завершении

1. Что сделано — по пунктам плана задачи.
2. Результаты верификации: build/tests/lizard/dogfood — конкретные числа
   («тесты: 412/412», «13/13 synthetic», «lizard: 0 warnings»).
3. Контрольные кейсы из задачи: ожидание vs факт, таблицей.
4. Что НЕ сделано и почему (если есть) — отдельным списком, без смешивания с готовым.
5. Ждать команды на коммит.
