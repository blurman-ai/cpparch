## Цель

Дать находкам дубликатов **меру важности (severity)**, основанную не на текстовом
сходстве в снапшоте, а на **согласованности правок копий во времени**. Сейчас код
меряет только структурное сходство (`weighted`/`lcs`/`line`) — это сходство, не
вред. Severity как вычисляемый выход в подсистеме **отсутствует** (grep
`priority|severity|importance` по `src/scan/duplication/` — ноль).

## Первопричина (литература)

Сильнейший эмпирический предиктор вреда копипаста — **inconsistent change**:
правят одну копию, забывают сёстры. Bettenburg et al. (WCRE'09): «почти каждое
*непреднамеренное* несогласованное изменение клона приводит к дефекту»
(«clone consistency-defect»). При этом ряд работ (Krinke; Harder & Göde) показал,
что клоны бывают стабильнее обычного кода — ⇒ вред **не универсален**, и текстовое
сходство его не предсказывает. Подробнее и со ссылками:
[../../docs/research/code_clones.md](../../docs/research/code_clones.md) §2.

Вывод: **важность клон-группы ≈ частота, с которой её копии менялись
рассогласованно по истории**, а не их Jaccard сегодня.

## Контекст: инфраструктура уже близка

`#054` (diff-mode) уже умеет **ходить по git-истории без checkout** и атрибутировать
копипасту коммиту. Это естественное место, где для клон-группы считается:

- сколько раз копии менялись в **одном** коммите (consistent co-change);
- сколько раз менялась **одна** копия, а сёстры — нет (inconsistent → risk);
- отношение inconsistent / total → severity-вес пары/группы.

То есть нужен не новый бэкенд, а надстройка над уже существующим обходом истории.

## Скоуп (предложение)

1. Клон-группа = транзитивное замыкание пар выше гейта (а не только пары) — нужна
   кластеризация пар в группы (сейчас отчёт — только пары; см. gap в research-доке §5.2).
2. По истории: для каждой группы посчитать co-change matrix → `inconsistencyRatio`.
3. Свернуть в severity-ярлык (HIGH/MED/LOW), **ортогональный** similarity-метрике
   (mem:`fp_classification_rules`: «similarity-score ≠ refactor-priority»).
4. Surface severity рядом с `type` в отчёте (тип уже выведен — см. #056 wire-in).

## Acceptance / fixtures

- Фикстура с историей: группа с consistent co-change → LOW; группа, где одну копию
  чинили, а сёстры протухли → HIGH.
- Детерминизм (CI-контракт): одинаковая история → одинаковый severity.

## Границы / риски

- Дорого: требует обхода истории — оправдано только в diff/baseline-режимах, не в
  snapshot. Держать за флагом, как и весь #054.
- Renamed/moved файлы по истории (P0.2 уже частично) — co-change легко
  переоценить/недооценить при rename; нужна follow-rename атрибуция.
- YAGNI-проверка: внедрять только если severity реально **меняет действие** автора
  (что чинить первым), а не как ещё одну цифру в отчёте.
