## Цель

Дать `archcheck --diff` точку входа для дубликатов: репортить копипаст,
**внесённый данным коммитом** (added-lines ∩ clone-spans), чтобы это работало
как PR-гейт в CI — а не только снимок всего дерева, как сейчас.

## Контекст

**Зачем:** copypaste и рост сложности — каноничные «плавные» деграданты
(constraint decay, ради чего проект): один дифф невинен, через год архитектура
сгнила. Сейчас в gating только graph-drift; копипаст невозможно гейтить, потому
что у детектора клонов **нет diff-входа**.

**Что есть (проверено по коду 2026-06-13):**
- ✅ Токеновый клон-детектор — `scanForDuplication` (shipped), но только снимок,
  зовётся из `src/cli/preview_commands.cpp:126`.
- ✅ git-diff плумбинг (old blob vs new blob, added-line ranges) — `src/cli/diff_command.cpp`,
  тот же путь, что кормит LCX.
- ✅ Шаблон «old vs new → что внесено» — `analysis/local_complexity/scan_commit.py`
  (match → reason → finding); per-commit для сложности уже написан.
- ❌ В `diff_command.cpp` НЕТ ни одного вызова duplication — `--diff` копипаст не
  репортит вообще. Это и есть гэп.

**Граница с #103:** #103 — корпусный research (Python, 185 реп, CSV, валидация
порогов). Эта задача — продуктовое правило в shipped C++, которое падает в CI.
#103 даёт пороги, эта даёт гейт.

## Алгоритм (минимальная рабочая версия — то, на чём сошлись)

1. `git diff parent..HEAD` → **диапазоны добавленных строк** по каждому файлу
   (плумбинг уже есть в diff_command).
2. Прогнать `scanForDuplication` на **новом** дереве → пары клонов (span A ↔ span B).
3. **Кандидат-находка** = пара, где хотя бы один span **пересекается с
   добавленными строками** коммита. Смысл: «дифф добавил код, являющийся клоном
   чего-то ещё».

**FP-guard (единственный реальный, см. ниже):** отбросить пары, которые **уже
были клонами в parent-дереве**. Случай: дубль X↔Y существовал до коммита, коммит
лишь задел одну копию (reformat) → её span пересёкся с added-lines → ложное «ты
внёс копипаст». Лечится сравнением с parent: пара «внесена», только если в parent
она не матчилась. Это второй скан (старого дерева) — дороже, но точнее.

**НЕ-FP (важно, чтобы не городить лишнего):** «перемещённый код» FP **не даёт**.
Если код реально переехал (удалён из A, добавлен в B) — в новом дереве он в одном
экземпляре, пары нет, ловить нечего. Защита от rename не нужна. Если обе копии
остались — это настоящая дупликация, и она *обязана* гореть.

## Валидация (двухступенчатая — цель пользователя)

Не реплеить всю историю реальной репы (дорого, тысячи ранов). Достаточно
**курируемого набора из 10 коммитов — 5 позитивов + 5 негативов**.

**Предусловие: сначала фича** (план выше) — без duplication-ветки в `diff_command.cpp`
любой прогон на копипаст покажет пустой отчёт.

**Ступень 1 — эмулируем CI локально.** Прогнать те же 10 коммитов через
`archcheck --diff parent..commit` локально (то же, что CI вызовет на runner'е).
Быстрый цикл на известном ground truth: сверить, что все 10 ведут себя как надо,
до всякого GitHub. Тут ловятся баги детектора, а не плумбинга.

**Ступень 2 — тестовая репа на GitHub.** Когда локально чисто — завести тестовый
репо, workflow на push-триггере (не PR), диапазон коммита через
`github.event.before..github.sha`:
```
archcheck --diff "${{ github.event.before }}..${{ github.sha }}" .
```
Каждый запушенный коммит = один CI-ран = один отчёт. Прогнать контрольные 10,
посмотреть как это выглядит вживую — красный чек / sticky-комментарий / step
summary. (Можно и через PR — тогда
[example_archcheck_pr.yml](../../.github/workflows/example_archcheck_pr.yml) как есть.)

**Набор коммитов** — подобран так, чтобы доказать и recall, и precision (негативы
бьют по FP-классам). Источник позитивов: реальные copy-paste сниппеты из OSS, но
коммиты собрать самим — правдоподобно + ground truth известен.

5 позитивов (обязаны гореть):
- функция скопирована в существующий файл (главный кейс Juergens)
- файл-копия добавлен целиком
- крупный блок скопирован
- ещё 2 вариации (EXACT / RENAMED)

5 негативов (обязаны молчать):
- уникальный новый код
- **перемещение** (код переехал — копии нет → move не FP)
- дубль ниже порога токенов
- **pre-existing клон, который коммит просто задел** (reformat одной копии) →
  проверяет parent-guard, самый важный негатив
- formatting-only изменение

**Критерий успеха:** все 10 ведут себя как надо на живой CI-поверхности (красный
чек/комментарий на 5, тишина на 5).

## Ключевые решения

| Решение | Причина |
|---------|---------|
| added-lines ∩ clone-spans | минимальный сигнал «коммит внёс клон», данные все есть |
| parent-guard от «задели старый дубль» | единственный реальный FP-класс |
| rename-guard НЕ делаем | move → копии нет → пары нет → FP не существует |
| advisory на старте, не gating | precision сначала на корпусе (#103), потом гейт |
| продукт ≠ #103 | #103 research/пороги; эта — shipped-правило в CI |
