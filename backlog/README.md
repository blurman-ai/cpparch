# backlog/

Каталог задач archcheck. Один файл = одна задача.

## Структура

```
backlog/
  new/*.md              — заведены, ещё не начаты, кандидаты для текущего релиза (v0.1 / v0.2)
  wip/*.md              — в работе (есть Дата старта)
  future/*.md           — заведены, но явно отложены в более поздний релиз (v0.3+) — design / contract задачи
  completed/*.md        — завершено (архив знаний)
  backlog_review.md     — последний отчёт /backlog-review (генерируется)
  TASK_TRACKER.md       — опциональный общий трекер
```

## Формат имени файла

```
NNN_<priority>_<short_name>.md
```

- **NNN** — 3-значный последовательный ID, никогда не меняется. Ссылка `#NNN` живёт вечно.
- **priority** — 3-буквенный код:
  - `blk` — blocker (нельзя сдвигаться без этого)
  - `crt` — critical (важно сейчас)
  - `maj` — major (значимо, но терпит)
  - `min` — minor (приятно иметь)
- **short_name** — snake_case, читаемое имя задачи.

Примеры: `004_blk_project_skeleton.md`, `006_maj_spec_refactor.md`.

Меняется приоритет → файл переименовывается (`maj` → `crt`). ID остаётся прежним, ссылки `#004` живы.

## Жизненный цикл задачи

1. **Создание** — `/create-task <snake_case_name>`. Файл — `backlog/new/NNN_<priority>_<name>.md`, статус `new`.
2. **Старт работы** — переместить в `backlog/wip/`, проставить `**Дата старта:** YYYY-MM-DD`, статус → `wip`. Для значительных задач — завести feature-ветку `<type>/<NNN>-<slug>` (см. [`docs/dev/git_workflow.md`](../docs/dev/git_workflow.md)). Для мелких — direct push в master.
3. **Прогресс** — `/checkpoint` обновляет файл.
4. **Коммиты** — `/commit` использует Conventional Commits и ссылается на задачу через `(#NNN)`. Hash коммита — в **Изменённые файлы**.
5. **Завершение** — переместить в `backlog/completed/`, статус → `done`, дописать секции **Как работает / Чем управляется / С чем связана / Диагностика**.
6. **Ревью** — `/backlog-review` периодически отлавливает протухшее (`new` без движения, `wip` висит долго).

## Зачем разделение new / wip / future

- **`new`** — очередь для текущего релиза. Берём отсюда. «Не начато» — нормальное состояние.
- **`wip`** — что-то реально делается *или* брошено. Если задача висит долго относительно `Дата старта` — явный сигнал «застряло».
- **`future`** — design / contract задачи, явно ожидающие более позднего релиза (v0.3+). Чтобы не путались в очереди `new/` и не шумели в `/backlog-review`. У такой задачи в шапке стоит `**Целевой релиз:** v0.X+`. Когда фаза подходит — переезжает в `new/`.

`new/` и `future/` различаются по timing, не по приоритету: в `future/` может лежать `blk`-задача — она блокер, но не для ближайшего релиза.

## Шаблон активной задачи

См. `/create-task` — создаёт файл с готовым каркасом.

## Ссылки между задачами

В полях `**Блокирует:**`, `**Заблокирован:**`, `**Related:**` ссылаться по ID:
```
**Блокирует:** #004 (project_skeleton), #002 (github_actions_ci)
```

Имя в скобках — для удобства чтения, ID — устойчивая часть. Если задача переехала в `wip/` или `completed/` — ID не меняется, ничего поправлять не надо.

## Принципы

- **Один файл — одна задача.** Если задача распухает — разбить, поставить `Related:`.
- **Задача без fixtures не существует** (если это правило). Чеклист fixtures есть в шаблоне.
- **Завершённые задачи — это документация системы**, а не свалка. Секция «Как работает» должна позволять понять решение без чтения кода.
- **Дата создания обязательна.** Это единственный способ отловить протухшее.
- **Дата старта обязательна при переезде в wip/.** Это способ отловить «висит без движения».
