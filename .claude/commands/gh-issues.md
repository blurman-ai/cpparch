Разобрать открытые GitHub issues этого репо в задачи бэклога.

Аргументов нет. Скилл тянет все **open** issues через `gh`, по каждому решает: создать новые задачи в `backlog/new/` или обновить уже существующие, потом комментирует issue списком созданного/обновлённого и закрывает.

## Инварианты

- **Открытый issue = ещё не разобран.** Закрытие — единственный маркер «обработано». Не вешать labels.
- **Папка для новых задач: `backlog/new/`.** Даже если задача очевидно горит — пусть пользователь сам двигает в `wip/`.
- **Никогда не коммитить.** Скилл только пишет файлы и работает с GitHub. Коммит — отдельно через `/commit`.
- **Спрашивать при сомнениях, не при каждом действии.** Безопасные случаи (новый issue без двойников, явная карта 1-к-1, чёткие подзадачи) — идти без подтверждения. Сомнительные (обновление существующей задачи, размытый скоуп, неполное покрытие issue задачами) — спрашивать.

## Шаги

### 1. Получить открытые issues

```
gh issue list --state open --json number,title,body,author,createdAt,labels --limit 50
```

Если пусто — сказать «открытых issues нет» и закончить.

### 2. По каждому issue — классифицировать

Прочитать `title + body`. Решить:

**Сколько задач рождает этот issue?**
- **N задач**, если в теле явный нумерованный список подзадач: маркеры `Issue 1 — …`, `## Issue 1`, `### 1.`, `1. **Title**` подряд, или таблица «находок». Пример: #6 (Arch audit) с `Issue 1 — README documents…` × 9.
- **1 задача** в остальных случаях, включая обычные нумерованные списки внутри плана (`1. шаг … 2. шаг …`) и issue-планы (#5 — у него уже расписан план фаз, это одна задача).
- Если сомневаешься — показать пользователю заголовки находок, спросить «N отдельных / одна rollup / гибрид».

**Связан ли issue с уже существующей задачей?**
- Прогрепать body на упоминания `#NNN` / `042_...` / имена файлов из `backlog/`.
- Если issue упоминает draft `NNN_*.md` — это и есть кандидат на обновление.
- Дополнительно: Grep по ключевым словам из заголовка по `backlog/new/`, `backlog/wip/`, `backlog/future/`, `backlog/completed/`. Хиты показать.

### 3. Для каждой находки — действие

**A. Создать новую задачу.** Условия: нет похожей в бэклоге, либо пользователь подтвердил «новая».

Следовать конвенции `/create-task`:
- ID = `max(NNN из backlog/{new,wip,future,completed}) + 1`, pad to 3 digits. На несколько новых задач из одного issue — выдавать подряд идущие ID.
- Имя: `NNN_<priority>_<slug>.md`, где `slug` — snake_case из заголовка находки.
- Приоритет — из issue: ищи слова `critical`, `major`, `minor`, `blocker` в теле находки. По умолчанию `minor`.
- Модуль — по содержимому: `DOCS` для документации, `RULES`/`SCAN`/`GRAPH`/`CONFIG`/`REPORT`/`CLI`/`BUILD` для кода, `BACKLOG` для гигиены бэклога.
- В шапке проставить `**Related:** #<issue_number> (gh)` чтобы был линк назад.
- В `## Контекст` — короткая выжимка из тела находки (1-3 абзаца), не копипастить целиком.
- В `## План выполнения` — конкретные шаги, если они есть в issue (например, «acceptance» секции). Если шагов нет — `- [ ] TODO: разбить на шаги`.

Шаблон файла — тот же что в `/create-task` (см. `.claude/commands/create-task.md`).

**B. Обновить существующую задачу.** Условия: пользователь подтвердил, что задача в бэклоге покрывает находку.

- Открыть файл, прочитать целиком.
- Показать пользователю: «issue #X добавляет такую-то инфу к задаче #Y. Что именно вписать?» — предложить конкретный патч (одна-две строки в `## Контекст` / новый пункт в `## План выполнения` / обновление `**Related:**`).
- Применить через Edit.
- В `**Related:**` дописать `#<issue_number> (gh)` если ещё нет.

**C. Пропустить.** Условия: issue — это вопрос/обсуждение/спам, никаких действий не требует.
- Спросить пользователя: «Закрыть как not-actionable, или оставить открытым?»

### 4. Прокомментировать и закрыть issue

После того как все находки issue обработаны:

```bash
gh issue comment <N> --body "$(cat <<'EOF'
Разобран в задачи бэклога:

- Создано: #NNN [title] — `backlog/new/NNN_..._....md`
- Обновлено: #MMM [title] — `backlog/new/MMM_..._....md` (добавлено: ...)

EOF
)"
```

Затем:
```bash
gh issue close <N>
```

**Когда спрашивать перед закрытием:**
- Найдено меньше задач, чем ожидалось из тела (часть находок не покрыта).
- Все действия — только апдейты существующих задач (issue не дал новой работы — может стоит оставить открытым?).
- Issue содержит вопрос пользователю или ожидает ответа.

В безопасном случае (созданы новые задачи, явно покрывают весь скоуп issue) — закрывать без подтверждения, комментарий уже всё объясняет.

### 5. Итог

Кратко в чат: сколько issues разобрано, сколько задач создано/обновлено, сколько issues закрыто, сколько осталось открытыми и почему.

## Когда останавливаться и спрашивать

- Issue упоминает существующую задачу, но содержимое расходится с тем, что в файле — спросить, какая версия истинна.
- Issue выглядит как rollup нескольких задач, но границы размыты — показать черновое разбиение.
- Нумерованный список выглядит как подзадачи, но это могут быть просто шаги одной задачи — спросить.
- Issue затрагивает скоуп v0.2+/`future/` — спросить, в `new/` или сразу в `future/`.
- При обновлении существующей задачи изменения большие (> 10 строк патча) — показать diff, спросить.

## Не делать

- Не двигать задачи между `new/` / `wip/` / `future/` (это работа `/issue`, `/checkpoint`, `/fix-issue`).
- Не править `pending/`. Никогда. См. memory `feedback_pending_folder`.
- Не закрывать issue без комментария со ссылками на задачи.
- Не коммитить созданные файлы — пользователь сам через `/commit`.
- Не вешать labels на GitHub.
