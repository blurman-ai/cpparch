# AGENTS.md

Руководство для coding agents, работающих в репозитории `cpparch`.

## Язык и стиль работы

- Отвечай пользователю по-русски.
- Пиши коротко, предметно и без лишней болтовни.
- Если документы конфликтуют, приоритет такой:
  `docs/architecture-spec.md` > `docs/MVP.md` > `README.md`.

## Состояние проекта

- Репозиторий пока в стадии **pre-implementation**.
- На текущий момент здесь нет `src/`, `CMakeLists.txt`, сборки и тестов.
- Репозиторий содержит в основном спецификации, backlog и служебные инструкции.
- Рабочее имя репозитория: `cpparch`.
- Основное имя инструмента в спецификации: `archcheck`.
- Вопрос финального имени ещё не закрыт. Не закрепляй брендинг без необходимости.

## Что это за продукт

`cpparch` / `archcheck` — это **CI-first CLI** для проверки архитектурных инвариантов в C++ проектах.

Инструмент должен:

- читать `compile_commands.json`;
- строить граф зависимостей по include и, позже, по AST;
- применять YAML-описанные модульные правила и дефолтные правила;
- сообщать нарушения в формате `file:line:column`;
- завершаться с ненулевым кодом при нарушениях.

Позиционирование принципиально важно:

- не "ArchUnit для C++";
- а **Lakos physical design + C++ Core Guidelines SF.* checks в CI**.

Сохраняй эту рамку в документации, сообщениях и описаниях фич.

## Что это не делает

Не размывай продукт за пределы этой области:

- не линтер;
- не bug finder;
- не formatter;
- не include optimizer;
- не GUI;
- не web dashboard;
- не IDE extension.

Если новая идея противоречит этому списку, сначала остановись и проверь, нужна ли она вообще.

## Архитектурные инварианты

Планируемый pipeline:

```text
parser → graph → rules → reporter
```

Планируемые подсистемы:

- `config/` — YAML loader и `Config`;
- `scan/` — два backend-а:
  `include_scanner` и `clang_scanner`;
- `graph/` — DAG, циклы, levelization, CCD/ACD/NCCD;
- `rules/` — правила по одному классу на файл;
- `report/` — text/json/sarif reporters.

Ключевые ограничения:

- Разделение на `include_scanner` и `clang_scanner` — **осознанное**. Не схлопывай их без явного обсуждения.
- **Одно правило = один класс = один файл.**
- Регистрация правил — через статическую таблицу.
- Не строй "универсальный rule engine", если задача этого прямо не требует.

## Технологические ограничения

- Язык: **C++20**.
- Сборка: **CMake**.
- Парсинг: **libclang/libtooling**.
- YAML: `ryml` или `yaml-cpp`.
- Тесты: Catch2 или GoogleTest.
- CI: GitHub Actions.

Ограничения по зависимостям:

- без Boost;
- без тяжёлых graph-библиотек;
- предпочитай простые структуры данных стандартной библиотеки.

Целевой дистрибутив:

- один статически собранный бинарь на платформу;
- плюс Docker image;
- runtime должен быть минимальным.

## MVP-границы

Для MVP не добавляй:

- AST-based rules;
- plugin system;
- visualization;
- auto architecture inference;
- IDE integrations.

Предпочтительный порядок бутстрапа:

1. reader для `compile_commands.json`
2. include graph
3. cycle detection
4. module mapping
5. dependency rules
6. CLI
7. text output
8. JSON output
9. fixtures

## Правила качества и проектирования

Базовые принципы:

- **YAGNI**
- **Authority over opinion**
- **Zero-config first**
- **Deterministic output**
- **Boring tech**

Практические правила:

- Сначала проверь, можно ли решить задачу удалением кода.
- Сначала поищи, нет ли уже нужной сущности, перед созданием новой.
- Не добавляй абстракции "на вырост".
- Не рефактори код, который не понимаешь.
- Не делай большой рефакторинг посреди фичи без отдельного согласования.

Жёсткие лимиты из `docs/code_quality.md`:

- функция: до 30 строк;
- класс: до 300 строк;
- параметров: до 4;
- вложенность: до 3 уровней;
- новые строки кода: до 50 на изменение, не считая тестов и fixtures;
- новых файлов: до 2;
- новых классов: до 1;
- новых абстракций: 0 без явного запроса.

## Правила C++ кода

Следуй `docs/code_style.md`. Ключевое:

- UTF-8 без BOM, Unix newlines;
- длина строки до 120;
- отступы: **3 пробела**;
- скобки: **Allman**;
- один публичный тип на файл;
- публичные методы: `PascalCase`;
- `private`/`protected` методы: `camelCase`;
- поля: `_name`;
- namespace: `lower_snake_case`;
- интерфейсы начинаются с `I`.

Предпочитай:

- `[[nodiscard]]`;
- `noexcept`, если функция не бросает;
- `std::string_view`, `std::span`;
- `std::optional`, `std::variant`;
- `std::ranges`;
- `concepts`;
- RAII и стандартную библиотеку.

Обязательно:

- без `using namespace` в заголовках;
- заголовки self-contained;
- `#pragma once` или include guards;
- без анонимных namespace в заголовках.

## Правила и fixtures

Для этого проекта правило без fixtures не существует.

Каждое новое правило обязано иметь:

- `fixtures/<rule>/pass/`
- `fixtures/<rule>/fail_*/`

Если фичу нельзя проверить через fixtures, её не надо реализовывать.

Каждое дефолтное правило должно иметь атрибуцию:

- C++ Core Guidelines;
- Lakos;
- Martin.

Если источника нет, это не дефолтное правило, а опциональное или отдельный lower-priority check.

## Dogfooding

`cpparch` обязан проверять сам себя.

Не допускаются изменения, которые ломают собственные правила проекта, включая:

- циклы;
- нарушения SF.7 / SF.8 / SF.9 / SF.11 / SF.21;
- архитектурные решения, противоречащие собственной модели модулей.

## Контракты CLI

Не меняй коды возврата без явного версионирования:

- `0` — OK
- `1` — violations found
- `2` — config / parsing error
- `3` — internal error

## Workflow

Задачи живут в `backlog/`, по одному `.md` на задачу:

- `backlog/new/` — заведено, но не начато;
- `backlog/wip/` — в работе;
- `backlog/completed/` — завершено и превращено в документацию.

Если работа ведётся по задаче, уважай структуру backlog и не смешивай несколько задач в один файл.

Для завершённых задач ожидается, что в файле будут заполнены секции:

- как работает;
- чем управляется;
- с чем связана;
- диагностика.

## Ограничения на действия агента

- Не коммить без явной просьбы пользователя.
- Не запускай сборку без явной просьбы пользователя.
- Когда сборка появится и пользователь попросит её запустить, предпочитай `Debug`, а не `Release`.
- Не придумывай лишнюю структуру проекта до появления реального кода и подтверждённой необходимости.

## Короткая проверка перед любым изменением

Перед тем как считать задачу выполненной, проверь:

- не расширил ли ты scope сверх MVP;
- не добавил ли ты абстракцию без явной пользы;
- можно ли удалить часть нового кода;
- есть ли проверяемый путь в fixtures или тестах;
- не противоречит ли изменение Lakos/Core Guidelines framing;
- не конфликтует ли изменение с `docs/architecture-spec.md`.
