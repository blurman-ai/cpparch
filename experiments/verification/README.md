# Verification corpus — данные глазной верификации дубликат-чекера

Дедикейтед, **трекаемая** папка под данные верификационных прогонов чекера дубликатов
(в отличие от `../ai_repo_run/`, который — scratch для воспроизводимых прогонов и почти
весь под `.gitignore`). Здесь лежит **невоспроизводимое**: размеченные глазами вердикты
и дистиллированный из них регресс-корпус.

Источник истины для задачи **#070** (FP-фиксы дубликат-чекера) и acceptance-замеров.

## Что здесь

| Файл / папка | Что это |
|---|---|
| `round2_verdicts/` | **Источник.** 135 JSON (по репе) — вердикты round-2 верификации (#066/#067): `confirmed` (TP) + `false_positives` (FP, с `fp_class`). LLM-eyeball разметка — **не regenerable**, держим в git как ценные данные. |
| `build_fp_corpus.py` | Генератор корпуса: склеивает `round2_verdicts/*.json` → детерминированный датасет. |
| `fp_corpus_r2.tsv` | **Корпус.** 340 строк (143 TP + 197 FP), по строке на находку: `repo, sha, label, fp_class, klass, added_loc, base_loc, note`. |
| `fp_corpus_r2_summary.json` | Агрегаты: precision 42.1%, разбивка FP по классам. |

## Зачем (что меряем)

Корпус — это **линейка для FP-гардов** задачи #070. Каждый новый фильтр меряется против
него: `precision_before/after`, `TP_kept/143`, `FP_removed/197`, `FP_removed_by_class`,
`new_FN`. Без размеченных данных тюнинг порогов = гадание (см. #070 §4).

## Ground truth = round 2

Берём round 2, т.к. это **единственный машиночитаемый** набор вердиктов. Round 1
(195 TP / 406 FP / precision ~32%) жил только в `../ai_repo_run/VERIFICATION_REPORT.md`
(markdown) и расходится с round 2 (143 TP / 197 FP / 42.1%; `coincidental` 99→1).
Подробности расхождения — в #070 §Контекст.

## Регенерация

```bash
python3 experiments/verification/build_fp_corpus.py
```
`.tsv`/`.json` — производные; правки вносить в `round2_verdicts/` или генератор, потом
перегенерировать. Сортировка детерминированная — diff покажет только реальные изменения.
